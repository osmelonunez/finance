# v3.4.0 - Data Robustness and UX Safety

## Goal

Make Finance safer to use after the v3.2/v3.3 loan and bank work by tightening validation, reducing accidental duplicate submissions, and making destructive actions clearer.

This release also polished the management and loan-detail screens while keeping the database changes concentrated in a single migration.

## Implemented Scope

### 1. Text Limits And Data Constraints

Text limits were standardized in backend validation, frontend validation, and database constraints.

Final limits:
- Record concept: 40 characters.
- Record comment: 500 characters.
- Category name: 40 characters.
- Category description: 500 characters.
- Payment method name: 40 characters.
- Bank name: 40 characters.
- Loan name: 40 characters.
- Loan description: 500 characters.
- Loan usage concept: 40 characters.
- Loan usage comment: 500 characters.

Implementation notes:
- Shared limits live in `backend/validators.py`.
- The UI uses `data-maxlength` or `maxlength` where appropriate.
- The database enforces the limits through `014_data_robustness_constraints.sql`.
- All v3.4 database checks were consolidated into `014_data_robustness_constraints.sql` because the extra migration files only existed in the test environment.

### 2. Safer Form Submission

Global form handling was added in `frontend/templates/base.html`.

Implemented behavior:
- Live required-field and max-length validation.
- Submit buttons are disabled for blocking validation errors.
- Valid submits disable buttons to reduce accidental double submission.
- Button text changes to context-aware labels such as Saving, Creating, Deleting, or Processing.
- External submit buttons that use the `form` attribute are supported.
- Validation works with editable table rows, including accounts/cards and banks.

Covered flows include:
- Add/edit movements.
- Duplicate movements.
- Add/edit loans.
- Register loan usage.
- Edit expense through modal confirmation.
- Category, bank, account/card, demo data, and destructive management forms.

### 3. Destructive Action Modals

Destructive actions now use consistent Bootstrap-style confirmation modals.

Implemented confirmations:
- Delete a movement from detail.
- Delete a loan usage.
- Delete a loan.
- Delete a category.
- Delete demo data.
- Delete banks when allowed.
- Delete accounts/cards when allowed.

Existing double-confirmation remains for database reset.

### 4. Audit Metadata

Detail views now show quiet audit information.

Implemented:
- Movement detail displays Created and Last edit.
- Loan detail displays Created and Last edit.
- The “updated” label was renamed to “Last edit” to better match the user-facing meaning.

### 5. Business Event Logging

Logging was expanded and normalized for user actions and demo data.

Implemented event areas:
- Records: create, update, duplicate, delete.
- Loans: create, update, status changes, delete, usage create/delete.
- Categories: create, update, delete, delete blocked.
- Banks: create, update, delete, delete blocked.
- Payment methods: create, update, delete, delete blocked.
- Demo data: seed and clear.

Demo data logs now include a `[DEMO DATA]` prefix and per-entity counts so the seed process can be used as a visible smoke test.

### 6. Loan And Analytics Safety

Implemented:
- Loans marked Paid or Cancelled are automatically excluded from analytics.
- The paid demo loan is excluded from analytics.
- Existing payments and history remain intact when exclusion is enabled.
- Loan usage validation was tightened.
- Loan usage required-field feedback was adjusted so the form stays aligned.
- Loan edit start date now uses the same monthly date picker as loan creation.
- Changing a loan start date still recalculates the remaining term behavior already implemented in v3.3.

### 7. Management UX Polish

Payment methods management was reorganized.

Implemented:
- The screen is separated into Banks, Accounts, and Cards.
- Accounts and Cards have separate creation forms.
- The type field is no longer manually selected in the UI; the section determines the type.
- Banks can be edited with the same action pattern as accounts/cards: Edit, Cancel, Save, Delete.
- Bank delete is blocked when the bank is used by accounts/cards or loans.
- Required-field feedback in management forms stays inside each panel without breaking alignment.
- Accounts/cards table validation supports external form controls.

### 8. Documentation And Versioning

Implemented:
- Version bumped to `3.4.0`.
- README and README.es updated with the new stable version.
- Roadmap updated with v3.4 and links to these detail files.
- The v3.4 notes were moved from `local-notes` into `docs/v3.4-planning` so they are part of the project documentation.

## Migration

Release migration:
- `014_data_robustness_constraints.sql`

Purpose:
- Adds database check constraints for final text-length limits.
- Uses `NOT VALID` constraints so existing databases can apply the migration without immediately validating historical rows.
